## 三、字段含义与强制规则
### 1. status 字段定义
item.status 含义：
- 0：未售出
- 1：已售出

### 2. 一致性规则
必须满足：
1. 每个商品最多只能被交易一次
2. orders 表中每个 item_id 只能出现一次
3. 若某商品出现在 orders 表中，则该商品 status 必须为 1
4. 若商品 status = 0，则该商品不能出现在 orders 表中

### 3. 外键逻辑关系
- item.seller_id 必须来自 user.user_id
- orders.buyer_id 必须来自 user.user_id
- orders.item_id 必须来自 item.item_id

## 四、作业要求
### （一）数据库定义
1. 创建数据库
2. 创建三张表（user、item、orders）
3. 设置主键
4. 设置必要的完整性约束（非空、外键等）

### （二）数据操作
1. 插入给定的全部初始数据
2. 插入一个新商品（自定义）
3. 修改某商品价格
4. 删除一个未售出的商品

要求：
- 操作必须真实作用于数据库
- 操作后刷新页面能看到结果变化

### （三）基本查询
系统必须实现以下查询，并通过页面交互展示结果：
1. 查询所有未售出的商品
2. 查询价格大于 30 的商品
3. 查询“生活用品”类商品
4. 查询 u001 发布的所有商品

### （四）连接查询
系统必须实现以下查询，并通过页面交互展示结果：
1. 查询所有已售商品及其买家姓名
2. 查询每个订单：商品名 + 买家名 + 日期
3. 查询卖家是 u001 的商品是否被购买

### （五）聚合与分组
必须展示：
1. 统计商品总数
2. 统计每类商品数量
3. 计算所有商品平均价格
4. 查询发布商品数量最多的用户

### （六）视图
创建视图：
1. 已售商品视图（包含商品名 + 买家ID）
2. 未售商品视图

### （七）简单“业务逻辑”实现
用 SQL 实现：“购买商品”操作，即当用户购买某商品时，必须完成：
1. 在 orders 表中新增一条记录
2. 修改 item 表中该商品 status 为 1

并保证：
- 已售商品不能再次购买

### （八）安全性
说明（文字即可）：
1. 如何防止普通用户删除数据
2. 如何限制用户只能查询数据

### （九）并发与恢复
说明（简单描述即可）：
1. 两个用户同时购买同一商品会出现什么问题
2. 如何解决（例如加锁/事务）
3. 如果系统崩溃，如何恢复订单数据
